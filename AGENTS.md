# AGENTS.md

## 역할 라우팅 규칙

| 역할 | 일 | 접점 |
| --- | --- | --- |
| 근거 역할 | 주장 → 출처강도 점검 | LinkNote RAG `/query`, `/study/claim` |
| 경계 역할 | 문구 검수 | `/study/audit` + 레지스트리 |
| 빌드 역할 | 코드 구현 | Cline/Gemini + AGENTS.md |
| 오케스트레이터 | 설계·리뷰·라우팅 | 전체 |

## 작업 시작 규칙

- 신규 작업은 항상 `AGENTS`, `SPEC`, `레지스트리`를 먼저 읽고 시작한다.
- 세 문서 중 하나라도 없거나 접근할 수 없으면, 없는 항목을 명시하고 확인 가능한 범위만 먼저 정리한다.
- 작업을 시작하기 전에 현재 요청이 어떤 역할에 해당하는지 먼저 분류한다.
- 여러 역할이 함께 필요하면 오케스트레이터가 순서를 정하고 결과를 통합한다.

## 라우팅 기본 원칙

1. 주장의 타당성, 출처, 근거 강도를 다루면 근거 역할을 우선 적용한다.
2. 문구 위험, 경계 위반 가능성, 표현 적절성을 다루면 경계 역할을 우선 적용한다.
3. 구현, 수정, 코드 생성이 중심이면 빌드 역할을 우선 적용한다.
4. 요청이 복합적이거나 모호하면 오케스트레이터가 먼저 범위를 정리하고 필요한 역할로 분기한다.

## 출력 원칙

- 결과에는 어떤 역할로 처리했는지 또는 어떤 순서로 라우팅했는지 짧게 드러낸다.
- 근거 역할 결과에는 가능하면 출처와 강도를 함께 적는다.
- 경계 역할 결과에는 문제 표현, 이유, 대체 표현을 함께 적는다.
- 빌드 역할 결과에는 변경 내용과 영향 범위를 분명히 적는다.
