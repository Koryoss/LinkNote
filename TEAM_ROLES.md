# LinkNote 팀 역할 구분

우리는 한 팀이다. 이 문서는 누가 무엇을 맡는지, 경계가 겹칠 때 어떻게 처리하는지를 정한다.
사람(Yoojin)이 방향을 정하고, 코덱스와 클로드가 각자 맡은 영역을 구현한다.

## 멤버

- **Yoojin** — 제품 방향과 우선순위 결정, 최종 검토.
- **Codex** — 코어 엔진 / 데이터 / 로직 담당.
- **Claude** — UI·표현 레이어 / 문서화 담당.

## 한 줄 원칙

**코덱스는 "데이터를 만들고", 클로드는 "그걸 보여준다."**
클로드가 데이터 구조(검색 결과 키 등)는 건드리지 않고 화면만 손대면 둘이 부딪히지 않는다.

## 클로드가 잘하는 역할

- **화면 표현** — 카드, 갤러리, Hero, 레이아웃, 색·스타일 (`app.py`의 `render_*` 부분)
- **문서** — 로드맵, 역할 정리, 설명서, 주석
- **빠른 UI 실험** — "이런 느낌?" 하고 바로 보여주기

코덱스는 엔진과 데이터(`rag.py`, 검색·embedding·ChromaDB)를 맡는다.

## 역할 구분

### Codex (코어 엔진 · 데이터 · 로직)

지금까지 핵심 코드를 주로 작성해 왔다. 아래 영역의 소유자다.

- `rag.py` — 검색, embedding, 프롬프트 규칙, 답변 생성, 개념 추출 로직
- `providers/` — Ollama / OpenAI / Hybrid provider
- `pdf_loader.py` — PDF 텍스트 추출
- ChromaDB 스키마, chunk 구조, 메타데이터 필드 정의
- `reset_db.py`, `inspect_db.py` 등 데이터 운영 스크립트
- 데이터 파이프라인 흐름(저장 → 검색 → 연결)

### Claude (UI · 표현 · 문서)

화면에 보이는 표현 레이어와 문서를 담당한다.

- `app.py`의 화면 구성과 표현(Streamlit 레이아웃, 카드, 갤러리, Hero, 스타일)
- "지식 갤러리" 방향의 UI 요소 추가 (전체 리디자인은 사람 승인 후)
- `PROJECT_ROADMAP.md`, `TEAM_ROLES.md` 등 문서 작성·정리
- 표시용 보조 데이터(예: `CONCEPT_MAP` 같은 화면 전용 상수)

### 경계가 겹치는 곳: `app.py`

`app.py`는 두 사람이 함께 만진다. 충돌을 막기 위한 규칙:

- **데이터를 가져오고 처리하는 함수**(`get_library_overview`, `answer_question` 호출 등)는 Codex 소유.
- **그 결과를 화면에 그리는 부분**(`render_*` 함수, HTML/CSS)은 Claude 소유.
- 한쪽 영역을 바꿔야 하면, 상대 영역의 입출력 형태(반환 dict 키 등)를 깨지 않는다.

## 핸드오프 규칙

- **데이터 구조가 바뀌면 Codex가 먼저** (chunk 키, 메타데이터 필드, 검색 결과 형태). 그 다음 Claude가 UI를 맞춘다.
- **화면만 바뀌면 Claude가** 처리하고 데이터 계층은 건드리지 않는다.
- 둘 다 필요한 작업은 "데이터 → 표현" 순서로 나눠서 진행한다.
- 서로의 소유 영역을 바꿔야 할 땐 먼저 알린다(주석 또는 커밋 메시지에 표시).

## 현재까지의 작업 기록

- **Codex**: 코어 RAG 엔진, ChromaDB 파이프라인, provider 구조, PDF 학습/검색/답변 로직 구현.
- **Claude (1차 UI 작업)**: Hero 문구, 지식 갤러리 개념 카드, 연결된 자료 카드, `PROJECT_ROADMAP.md`의 UI Vision 섹션 추가. 전체 리디자인은 아직 하지 않음.
